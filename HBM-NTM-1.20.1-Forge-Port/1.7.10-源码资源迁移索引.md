## 迁移总览

1.7.10 到 1.20.1 不是 API 小改，而是平台级迁移。建议先把系统拆成可验证模块，而不是直接整体搬运。

优先级建议：

1. 建立 1.20.1 新工程骨架：mod id、注册、配置、资源包、数据生成。
2. 先迁移物品、方块、创造栏、语言、基础贴图。
3. 再迁移方块实体、菜单、网络包、配方和数据。
4. 最后处理渲染、OBJ 模型、复杂机器、核反应堆、爆炸、辐射、世界生成和兼容层。

高风险迁移点：

- `GameRegistry`、`@SidedProxy`、`cpw.mods.fml.*` 全部需要替换成现代 Forge/NeoForge 事件总线和 `DeferredRegister` 风格。
- 1.7.10 metadata/damage 物品和方块状态需要重建为 `BlockState`、独立物品、NBT/DataComponent 或 capability。
- `TileEntity` 需要迁移为 `BlockEntity`，GUI 需要迁移为 `Menu` + `Screen`。
- 1.7.10 旧资源路径 `textures/blocks`、`textures/items` 需要改到 1.20.1 的 block/item model JSON 体系。
- `IExtendedEntityProperties`、旧网络包、旧粒子和旧渲染注册都需要重写。

## 入口与生命周期

核心入口：

- `src/main/java/com/hbm/main/MainRegistry.java`
  - `@Mod(modid = RefStrings.MODID, name = RefStrings.NAME, version = RefStrings.VERSION)`
  - 旧式 `@SidedProxy`
  - 负责 preInit/init/postInit/server 事件、配置、注册、兼容、网络和事件处理器挂载

关键生命周期：

- `PreLoad(FMLPreInitializationEvent)`：配置、流体、方块、物品、世界生成、燃料、卫星、配方、危险属性、矿辞、GUI、TileEntity、实体映射、战利品、分发器、兼容预初始化
- `load(FMLInitializationEvent)`：成就、统计、IMC handler、NEI 兼容
- `PostLoad(FMLPostInitializationEvent)`：流体容器、机器配方、JSON 配方、砧配方、辐射/掉落/物品池配置、盔甲和防护注册、OC/AE2 兼容、世界矿层
- `preInit(FMLPreInitializationEvent)`：事件处理器、网络包、辐射和污染系统注册

现代迁移对应：

- `MainRegistry` 拆分为：主 mod 类、注册类、配置类、common setup/client setup、事件订阅类。
- `@SidedProxy` 替换为 `DistExecutor` 或 client-only event subscriber。
- `FMLPreInitializationEvent/FMLInitializationEvent/FMLPostInitializationEvent` 替换为 mod event bus 的 setup/enqueue work/register events。

## 注册系统索引

方块：

- `src/main/java/com/hbm/blocks/ModBlocks.java`
- 入口：`ModBlocks.mainRegistry()`
- 旧流程：`initializeBlock()` 后 `registerBlock()`
- 迁移：改为 `DeferredRegister<Block>` 和 `RegistryObject<Block>`，ItemBlock 也需要独立注册。

物品：

- `src/main/java/com/hbm/items/ModItems.java`
- 入口：`ModItems.mainRegistry()`
- 大量静态 `Item` 字段，许多功能由 metadata/enums 区分。
- 迁移：优先清点 metadata 物品，决定拆成多个 item，或改用 NBT/DataComponent/自定义模型谓词。

方块实体：

- `src/main/java/com/hbm/tileentity/TileMappings.java`
- 入口：`TileMappings.writeMappings()`
- 旧流程：`GameRegistry.registerTileEntity(...)`
- 迁移：改为 `BlockEntityType` 注册。旧字符串 ID 要保留迁移映射，避免老存档完全丢失。

实体：

- `src/main/java/com/hbm/entity/EntityMappings.java`
- 迁移：改为 `EntityType` 注册、属性注册、spawn placement、renderer 注册分离。

网络：

- `src/main/java/com/hbm/packet/PacketDispatcher.java`
- `src/main/java/com/hbm/main/NetworkHandler.java`
- 旧式 `IMessage/IMessageHandler` 和自定义 codec。
- 迁移：用现代 `SimpleChannel` 或 1.20.1 推荐网络 API，服务端处理必须显式切回主线程。

GUI：

- `src/main/java/com/hbm/main/GUIHandler.java`
- `src/main/java/com/hbm/inventory/container`
- `src/main/java/com/hbm/inventory/gui`
- 迁移：`IGuiHandler` 改为 `MenuType`、`AbstractContainerMenu`、client `Screen`。

配置：

- `src/main/java/com/hbm/config`
- 重点类：`GeneralConfig`、`WorldConfig`、`MachineConfig`、`RadiationConfig`、`WeaponConfig`、`CustomMachineConfigJSON`
- 迁移：普通配置进入 `ForgeConfigSpec`；大型机器/配方配置保留 JSON 或迁入 datapack。

## 建议的迁移工作包

第一阶段：可启动空壳

- 新建 1.20.1 Forge/NeoForge 工程。
- 注册 `hbm` mod id、基础配置、日志、creative tab。
- 搬迁语言 JSON 和少量测试物品/方块。

第二阶段：基础注册和数据资产

- 从 `ModItems`、`ModBlocks` 生成迁移清单。
- 为每个基础方块生成 `blockstates`、`models/block`、`models/item`。
- 把旧 `textures/items`、`textures/blocks` 分类迁到新路径。
- 用 tag 替代 OreDictionary。

第三阶段：机器最小闭环

- 选择一个简单机器做样板，例如单方块、有库存、有 GUI、有配方的机器。
- 建立 `BlockEntity`、`Menu`、`Screen`、同步包、RecipeType 模板。
- 再批量迁移同类机器。

第四阶段：复杂系统

- RBMK/PWR/fusion、导弹、爆炸、辐射、污染、世界生成分别建独立分支或模块迁移。
- 每个系统先迁移数据结构和保存格式，再迁移交互和渲染。

第五阶段：兼容和优化

- 逐步恢复 JEI/REI/EMI、AE2、OpenComputers 或替代兼容。
- 对爆炸、辐射、网络同步做性能审查。

## 当前扫描结论

这个 1.7.10 项目内容量很大，但入口集中度高。最适合的迁移策略是先把注册表和资源资产清单数据化，再用一两个简单机器建立 1.20.1 的现代样板。不要一开始迁移 `render`、`explosion`、`rbmk` 和 `worldgen`，这些模块 API 断层最大，适合作为后续独立攻坚项。
